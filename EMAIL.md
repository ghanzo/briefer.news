# EMAIL.md — Self-built email infrastructure

> **Status: in progress.** Steps 1-2 shipped 2026-05-26. Steps 3-8
> pending. Buttondown stays live in parallel until self-built is fully
> tested; canceled once first real send to a non-test subscriber works.

The self-built email path: AWS SES for delivery + postgres for subscriber
state + custom HTML template matching the site's design + signup /
unsubscribe flows we control end-to-end.

**Why self-built over Buttondown free tier:** brand control. Free tier
forces a "Sent via Buttondown" footer + the forward-to-subscribe widget
+ limited theming. Self-built ships pixel-perfect emails with the site's
exact typography, palette (sepia / cream / ink), and structure. AWS SES
is ~$0.10 per 1,000 sends — pennies at our scale, $30/mo even at 10K
subscribers.

---

## Architecture

```
signup form (site footer)
       │
       ▼
POST /api/subscribe (Lambda or local handler)
       │
       ▼
email_subscribers table (postgres, on Mac mini)
       │
       ▼
confirmation email (SES)
       │
   user clicks confirmation link
       │
       ▼
status: 'pending' → 'confirmed'
       │
       ▼
daily-send LaunchAgent at 08:30 PT
       │
       ▼
SES → subscriber inbox
       │
       ▼ (on bounce / complaint)
SNS → bounce handler → status: 'confirmed' → 'bounced'
```

### Postgres `email_subscribers` table

Schema in `pipeline/db/migrations/004_email_subscribers.sql`. Columns:
`id, email (unique), status (pending/confirmed/unsubscribed/bounced),
confirmation_token, confirmed_at, unsubscribed_at, unsubscribe_token,
edition_preference (us/china/both), notes, created_at, updated_at`. Auto-
touch trigger on update.

Python helpers in `scripts/email_subscribers.py`:
```
python3 scripts/email_subscribers.py add foo@bar.com [us|china|both]
python3 scripts/email_subscribers.py confirm <confirmation_token>
python3 scripts/email_subscribers.py unsubscribe <unsubscribe_token>
python3 scripts/email_subscribers.py get foo@bar.com
python3 scripts/email_subscribers.py list [us|china]
python3 scripts/email_subscribers.py counts
```

Importable from other scripts as `from email_subscribers import …`.

## What I should NOT do

- **Don't send a real email until SES is in production AND the bounce
  handler is wired.** A bounce in sandbox mode is silent; in production
  without a handler, repeated bounces hurt sender reputation.
- **Don't seed-import the Buttondown subscriber** (just `ghanzo@gmail.com`
  for now anyway) until double-opt-in is wired — would skip the consent
  step and create a CAN-SPAM exposure for any future subscribers.

## Cost guards (Step 8)

Once shipped, the daily-send wrapper enforces:
- **Daily send cap** — default 5,000/day; aborts above this, logs to
  morning brief. Adjust upward as needed.
- **CloudWatch billing alarm** — alerts at $5/mo, $20/mo, $100/mo
  thresholds. Notification target: ghanzo@gmail.com.
- **Kill switch** — `EMAIL_ENABLED=true|false` in `.env`. Daily-send
  wrapper checks and refuses to send if false. Useful for "halt all
  outbound" in case of weird behavior.
